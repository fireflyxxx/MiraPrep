## 1. 产品概述

### 1.1 产品定位

MiraPrep 是一款基于大语言模型的仿真模拟面试产品。用户上传简历后，AI 面试官基于简历内容、目标岗位和用户补充信息，进行一轮贴近真实场景的完整面试（支持语音与文字交流），并在面试结束后输出评级与逐题分析报告，帮助求职者发现短板、迭代表现。

### 1.2 核心价值

- **仿真性**：面试官具备真实面试的节奏——开场寒暄、简历深挖、技术/业务考察、追问、反问环节、收尾，而非机械问答。
- **个性化**：问题从用户简历和目标岗位生成，追问基于用户上一轮回答动态展开。
- **可复盘**：每次面试沉淀为结构化报告（逐题回答、参考答案、耗时、改进建议），形成成长记录。

### 1.3 目标用户

| 用户画像 | 场景 |
|---|---|
| 应届生/实习生 | 缺乏面试经验，需要低成本高频次演练 |
| 社招跳槽者 | 针对特定目标公司/岗位做冲刺准备 |
| 转行者 | 检验新技术栈掌握程度，暴露知识盲区 |

### 1.4 版本范围

- **当前前端（M1 基线）**：落地页、登录/注册、初次引导、工作台、我的面试、题库训练（占位）、面试配置向导、面试进行（**纯文字为主，语音为占位 UI**）、评级结果、面试报告——10 个页面已实现视觉与前端交互，数据暂为本地 mock，未接后端。
- **规划中（M2/M3）**：深色模式、语音 ASR/TTS、五维能力雷达图、评级揭晓仪式动效、落地页 FAQ/报告样例、报告导出/分享/历史对比、重练此题、第三方登录、个人设置页等（见第 7 章里程碑）。
- **更远期（Backlog）**：多语言/英文面试、面试官人设商店、团队版、模拟群面/系统设计白板、题库社区、移动端 App。

## 2. 整体用户流程

```
落地页(/) ──「登录/免费开始」──► 登录注册页(/auth)
                                      │
                        ┌─────登录成功─┴──注册成功────┐
                        ▼ 老用户                     ▼ 新用户
                 个人工作台(/dashboard)      初次引导页(/onboarding)
                        ▲                            │ 完成/跳过
                        └────────────────────────────┘
                        │ 点击「开始准备」
                        ▼
              面试配置向导(/interview/setup，3 步 Stepper)
          ① 上传/选择简历 → ② 岗位·难度·时长 → ③ 补充要求
                        │ 「开始面试」+ 大纲生成过场
                        ▼
              面试进行页(/interview/[sessionId]，语音/文字)
                        │ 面试结束
                        ▼
              评级结果页(/interview/[sessionId]/result)
                        │ 「查看完整报告」
                        ▼
              面试报告页(/report/[sessionId])
                        │ 「返回工作台 / 再练一场」
                        ▼
                 个人工作台(/dashboard)
```

## 3. 页面功能需求

### 3.1 落地页（`/`）

**目标**：3 秒内让访客理解产品是什么，引导注册转化。

**已实现结构（自上而下）**

1. **导航栏**：左侧 `M` 方形图标 + `MiraPrep.` 文字；中部锚点导航（功能 → `#features` / 工作原理 → `#how` / 常见问题 → `#faq`）；右侧「登录」（ghost）+「免费开始」（深色主按钮）。滚动超过阈值时导航栏切换为半透明毛玻璃背景（`backdrop-blur`）+ 细底边 + 轻阴影。
2. **Hero 区**（左右两栏）：
   - 左：主标题「像真实面试一样，练到你拿下 **offer**」（offer 为橙色 Space Grotesk）+ 副标题；主 CTA「开始一场面试 →」（橙色，hover 上浮）+ 次 CTA「观看演示」（描边）；下方三个头像叠加 + 「已帮助 **12,800+** 位候选人准备面试」。
   - 右：**面试对话模拟卡片**（浮动动画 `mira-float`）——展示 Mira 面试官头衔、一条面试官提问气泡、一条用户回答气泡、一条「正在聆听你的回答…」的声纹波形条，营造真实对话感。
3. **题库方向条**：小标题「覆盖主流方向的面试题库」+ 一排 pill 标签（前端工程 / 后端开发 / 算法 / 产品经理 / 数据分析 / 数据科学 / 运营 / 市场）。
4. **功能特性区**（`#features`）：`WHY MIRAPREP` 标签 + 标题「不是刷题，是真的把面试完整走一遍」+ **3 张等宽卡片**：会追问的面试官 / 基于你的简历定制 / 结构化评估报告。卡片 hover 上浮 + 橙色边框高亮 + 投影。
5. **工作原理区**（`#how`）：`HOW IT WORKS` 标签 + 标题「四步，开始你的面试」+ 右上「立即开始」按钮；**4 步横向流程**（01 上传简历 / 02 选择岗位与要求 / 03 进行仿真面试 / 04 获取评级报告），首步顶部为深色描边强调。
6. **深色 CTA 区块**：黑底卡片 + 橙色径向光晕，标题「下一场面试，别再靠临场发挥。」+ 副文案 +「免费开始 →」。
7. **Footer**（`#faq` 锚点当前指向此处）：`MiraPrep.` + 「© 2026 MiraPrep Labs · 让每一次面试都不慌」。

**已实现交互**
- 页面进入与页面间切换使用 View Transitions（§4.5）。
- 导航锚点平滑滚动（`scroll-behavior: smooth`）。

**未来规划**

### 3.6 面试进行页（`/interview/[sessionId]`）

**已实现整体**：沉浸式、隐藏全局侧边栏、浅色主题（白底）。

- **顶部状态栏**：`MiraPrep.` + 岗位（前端工程师·中级）+ 「第 N 题 · 进行中」+ 计时（脉冲点 + `14:32`）+ 「↩ 回看」+ 「结束面试」。

- **中央面试官区（单题聚焦式）**：声波球（径向渐变圆 + 橙色跳动波形 + 外圈脉冲环）；「MIRA 面试官正在提问 · 问题 0X」；大号问题文本；提示 pill（提示 + 该题引导语）。**当前为「一次一题、居中呈现」，非聊天流。**
- **底部输入区**：语音 / 文字模式切换。
  - **文字模式**：多行输入框（「输入你的回答，Shift + Enter 换行」）+ 「提交回答 · 继续」+ 辅助说明。
  - **语音模式**：声纹波形 + 圆形录音按钮（播放/停止切换）+ 「提交回答 · 继续 →」（录音后可用）+ 实时转写文案（当前为占位模拟）。
- **「回看」抽屉**：右侧滑出，列出本场已回答的问答（题号 + 问题 + 你的回答）；面试进行中可随时打开。
- **结束**：点击「结束面试」或答完最后一题 → 跳评级结果页（View Transition `nav-reveal`）。

**未来规划**
- **面试阶段指示**（自我介绍 → 简历深挖 → 专业考察 → 反问 → 收尾）随后端 `phase_change` 高亮。
- 面试官问题**流式打字 + TTS 播报**；「正在思考」三点占位；聊天流式历史（面试官左/用户右）+ 代码块渲染 + 自动滚动 + 「回到最新」。

## 4. 视觉与交互设计规范

> 以下为**当前已实现**的设计口径；深色模式、Framer Motion 动效、雷达图等为规划项。

### 4.1 设计基调

现代、扁平、克制的高级感。参考基准：**Linear**（排版节奏、微动效）、**Vercel**（极简黑白灰 + 单一强调色、清晰层级）、**Stripe**（落地页叙事、渐变克制运用）。原则：大量留白、白/浅灰底色 + 单一高饱和橙色强调、层级靠字重与间距而非线框、动效轻且快。

### 4.2 色彩系统

当前实现为**浅色单主题**，橙色为品牌强调色（非 PRD 早期设想的靛蓝紫）。

| 用途 | 色值 | 说明 |
|---|---|---|
| 品牌强调 `primary` | `#f97316` | 主 CTA、高亮、评级、图标；hover 深化 `#c2410c` |
| 强调浅底 | `#fff5ee` / `#ffe0cc` | 橙色语气的浅色面板、选中态；其上文字 `#9a3412` / `#7c2d12` |
| 链接 | `#ea580c` → hover `#c2410c` | 全局链接色（`@layer base`，可被按钮工具类覆盖） |
| 页面底色 | `#ffffff` / `#fafafa` / `#f7f7f7` | 白 / 应用内浅灰 / 引导页背景 |
| 卡片/面板 | `#ffffff` | 描边 `#eee` / `#e5e5e5` |
| 主文本 / 次文本 / 弱文本 | `#0a0a0a` / `#525252`·`#737373` / `#a3a3a3` | 深色近黑用于主文本与深色卡背景 |
| 语义 | 成功 `#16a34a`、成功浅底 `#f0fdf4` | 「已完成」「优秀」等 |

**评级配色（当前）**：综合评级以字母 + 加减呈现（A- / B+ / B…），主色橙 `#f97316`；逐题定性评级 `优秀` = 绿 `#16a34a`、`良好` = 橙 `#f97316`、`待提升` = 灰 `#a3a3a3`。
- 渐变仅用于：Hero 光晕、深色 CTA/工作台卡的径向光晕、评级环形。正文界面纯色扁平。

**未来规划**：接入 `next-themes` 深色/浅色双主题（跟随系统 + 手动切换），面试进行页默认深色；语义 token 化（`background/surface/border/foreground/muted/primary/success/warning/danger` + `grade-*`）。

### 4.3 字体与排版

- **展示/西文/数字/Logo**：`Space Grotesk`（权重 400–700）——用于标题数字、评级、计时、品牌字。
- **中文正文**：`Noto Sans SC`（权重 400/500/700）。
- 字号阶梯（实现值）：Hero display 42–60、h1 26–30、h2 32–40（落地页区块）、h3 16–18、正文 14–15、辅助 13、标签 12–12.5；行高正文约 1.6、标题约 1.1–1.2。
- 数字型数据（评级、计时、统计）建议 `tabular-nums`（规划中统一补齐）。

### 4.4 布局与间距

- 间距基于 4px 网格；卡片内边距约 22–30px；落地页区块间距 96–110px、应用内 24–44px。
- 圆角：大卡 18–26px、常规卡 12–20px、按钮 9–12px、输入框 10–14px、徽章/pill full。
- 阴影极轻，层级主要靠 `border` + 背景色差；浮层用较重阴影 + 遮罩 `rgba(10,10,10,.35)` + `backdrop-blur`。
- 内容最大宽度：落地页 1180px、工作台/应用内约 1000px、报告正文 840px、配置向导 720px、登录表单 380px。

### 4.5 动效规范

**当前实现**
- **页面间过渡**：Next.js **View Transitions API**（`experimental.viewTransition`），按页面对定制方向语义——`nav-forward`（深入）/`nav-back`（返回）/`nav-modal-in`·`nav-modal-out`（引导浮层进出）/`nav-reveal`（面试→评级揭晓）/`nav-fade`（兜底淡入）。
- **页面内动效**：CSS 关键帧——`mira-float`（Hero 卡浮动）、`mira-pulse`（脉冲）、`mira-bar`（声纹波形）、`mira-rise`（元素上浮进入）。
- 按钮 hover 亮度/位移、卡片 hover 上浮 + 边框高亮。
- 已遵循 `prefers-reduced-motion`：减弱动效时过渡时长归零、位移动画降级。

**未来规划**：引入 **Framer Motion** 统一页面内元素动效（进场 stagger、数字滚动、流式打字光标、评级揭晓仪式 600–900ms、`spring(300,30)`），与 View Transitions 分工（前者管页面内元素、后者管页面间）。

### 4.6 组件库与实现约定

**当前实现**
- 组件为手写 Tailwind + 少量共享组件（如 `Logo`、`LandingNav`、`RouteTransition`、`DashboardShell`——工作台/我的面试/题库训练共用的侧边栏 + 用户菜单外壳）；图标使用内联 SVG（含多变体镜面 Logo 图标）。
- 样式优先 Tailwind 工具类，复用值走 `globals.css` 的 `@theme` 与自定义动画类。
- 响应式：核心页面已做桌面布局，部分做了移动端栅格降级。

**未来规划**
- 接入 **shadcn/ui**（Button/Input/Dialog/Tabs/Select/Toast/Skeleton/Card…）+ `lucide-react` 图标；**Recharts** 图表（雷达/折线/sparkline）。
- 异步内容统一 **Skeleton 占位**（禁止 CLS）；统一空态组件（插画 + 标题 + 描述 + CTA）；Toast 右上角自动消失。
- 响应式断点 `sm 640 / md 768 / lg 1024 / xl 1280`，配置/面试/报告完整适配移动端（面试页移动端输入区底部全宽、语音按钮加大）。

## 5. AI 面试官 Agent 设计（FastAPI 侧）

### 5.1 面试流程状态机

```
GREETING（开场寒暄+流程说明）
   ▼
SELF_INTRO（自我介绍，1 题）
   ▼
RESUME_DEEP_DIVE（简历/项目深挖，2–4 题，每题 0–3 层追问）
   ▼
DOMAIN_ASSESSMENT（专业能力考察：基础知识/场景设计/编码思路，随难度与面试类型配置，3–6 题）
   ▼
BEHAVIORAL（行为面/HR 问题，0–2 题，HR 面类型时为主体）
   ▼
CANDIDATE_QA（反问环节：邀请用户提问并作答）
   ▼
CLOSING（收尾致谢，说明报告生成）
```

- 各阶段题量随「面试时长」配置动态伸缩；计时到点时状态机强制推进到 CANDIDATE_QA。
- **题目规划**：面试开始前（配置确认后的过场阶段），Agent 基于简历解析结构 + 岗位 + JD + 补充要求生成「面试大纲」（题目池 + 考察点映射），面试中按大纲执行并允许动态偏移。

### 5.2 追问与对话策略

- 每轮回答后 Agent 做即时评估（完整性/深度/真实性信号），据此决定：追问（回答有亮点或疑点）、给提示（明显卡壳）、换题（多次无效回答）。
- 追问深度上限 3 层，避免死磕；「高压追问型」风格提高追问概率与尖锐度，「温和引导型」增加提示与鼓励。
- Agent 需处理的边界情况：用户答非所问（礼貌拉回）、用户反问问题本身（澄清但不给答案）、用户长时间沉默（语音模式下主动询问）、不当内容（专业地终止话题）。
- 面试全程 Agent **不当场评价对错、不打分**，保持真实面试官口吻；所有评价延迟到报告。

### 5.3 评分与评级体系

- **单题评分**（0–10）：由评估模型在面试结束后统一批改，维度权重随题型变化（知识准确性、深度、结构化表达、结合自身经验）。报告展示时可映射为定性档（`优秀` ≥ 8 / `良好` 6–7 / `待提升` < 6），与前端逐题卡一致。
- **能力维度分**（0–100）：由逐题得分按考察点映射聚合。当前视觉稿仍展示四维 mock；M1 API 与联调统一冻结为五维：专业知识、项目深度、表达逻辑、临场应变、岗位匹配度，对应键 `professionalKnowledge/projectDepth/communicationLogic/adaptability/jobFit`。后端 `dimension_scores` 以 JSON 存储，前端负责中文标签映射，不再并行维护另一套四维接口模型。
- **总分** = 各维度加权（权重随面试类型配置）；**综合评级**以字母 + 加减呈现（S/A/B/C/D，可含 +/-），映射：S ≥ 90，A 80–89，B 70–79，C 60–69，D < 60（加减为区间内细分）。
- **参考答案生成**：结合题目考察点 + 用户简历背景生成（参考答案应贴合用户自身项目，而非通用答案）。
- 未完成面试：已答部分正常批改，报告标注「部分完成」，不计入评级统计趋势。

### 5.4 语音链路

- **ASR**：前端采集音频（MediaRecorder，16kHz）→ WebSocket 流式推送 FastAPI → 流式转写实时回显（可接入 Whisper 或云厂商流式 ASR）。
- **TTS**：面试官文本流式合成播报，句级切分以降低首音延迟；目标首字延迟 < 1.5s。
- 转写文本发送前允许用户编辑；原始音频保留用于报告回放（可在设置中关闭存储）。

> 语音链路为 **M2** 范围；当前前端面试页的语音模式为占位 UI（波形/录音按钮/转写文案为模拟），文字模式为主链路。

## 6. 系统架构与接口设计

### 6.1 总体架构

```
Next.js 前端 ──REST──► Spring Boot 业务服务 ──► MySQL / Redis / 对象存储(OSS/S3)
     │                        │
     │                        └─内部调用（REST/gRPC）─► FastAPI AI 服务
     │                                                    │
     └────WebSocket/SSE（面试会话、语音流）────────────────┘
                                              FastAPI ──► LLM API / ASR / TTS / 向量库
```

**职责划分**

| 服务 | 职责 |
|---|---|
| Spring Boot | 用户认证（JWT + Refresh Token）、用户资料与偏好、简历文件管理（上传至对象存储）、面试会话生命周期与记录、报告持久化与查询、统计聚合 |
| FastAPI | 简历解析（PDF/DOCX → 结构化 JSON）、面试大纲生成、面试官对话推理（流式）、ASR/TTS 网关、批改与报告内容生成 |

### 6.2 核心数据模型

```
User(id, email, password_hash, avatar, created_at, is_first_login)
UserProfile(user_id, job_direction, tech_stacks[], experience_level, status, preferences...)
Resume(id, user_id, file_url, file_name, parsed_json, is_default, created_at)
InterviewSession(id, user_id, resume_id, job_direction, job_title, jd_text,
                 difficulty, types[], duration_min, custom_requirements,
                 interviewer_style, status[created|ongoing|completed|aborted],
                 grading_status[none|pending|ready|failed], grading_error,
                 started_at, ended_at)
InterviewMessage(id, session_id, role[interviewer|candidate], content,
                 audio_url?, phase, question_id?, created_at)
Question(id, session_id, phase, text, focus_points[], order,
         think_seconds, answer_seconds, suggested_seconds, skipped)
Report(id, session_id, grade, total_score, dimension_scores{}, summary,
       highlights[], weaknesses[], created_at)
QuestionReview(id, report_id, question_id, score, reference_answer,

```

### 6.3 API 设计（Spring Boot，`/api/v1`）

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/auth/register` `/auth/login` `/auth/refresh` `/auth/send-code` | 认证 |
| GET/PUT | `/users/me` `/users/me/profile` | 资料与 Onboarding 配置 |
| POST | `/resumes`（multipart 上传，触发 FastAPI 解析） | 简历上传 |
| GET/DELETE/PATCH | `/resumes` `/resumes/{id}` | 简历库管理 |
| POST | `/interviews`（body=配置向导结果，返回 sessionId，异步生成大纲） | 创建面试 |
| GET | `/interviews/{id}/status` | 大纲准备状态（前端过场页轮询） |
| POST | `/interviews/{id}/end` | 手动结束 |
| GET | `/interviews?page&size&status?` | 面试记录分页列表（含会话/报告状态） |
| GET | `/reports/{sessionId}` | 报告详情 |
| GET | `/reports/{sessionId}/export` | PDF 导出 |
| GET | `/stats/overview` | 工作台区统计、加权综合评级与五维均值 |

> 各接口请求/响应的完整 JSON 契约以 `docs/tasks/` 对应任务文件为准（如 T-010 认证、T-020 简历、T-030 会话、T-051 报告/统计）。

### 6.4 实时接口（FastAPI）

| 协议 | 路径 | 说明 |
|---|---|---|
| WebSocket | `/ws/interview/{sessionId}` | 双向：用户文本/音频帧 ↑，面试官流式文本 token、TTS 音频帧、阶段变更事件、ASR 部分转写 ↓ |
| SSE（文字模式） | `/interviews/{id}/stream` | 纯文字面试用 SSE，M1 主链路 |
| POST | `/internal/resumes/parse` | Spring Boot 内部调用，简历解析 |
| POST | `/internal/interviews/{id}/grade` | 面试结束触发批改，完成后回调 Spring Boot 落库 |

消息协议采用 JSON envelope：`{type: "token"|"audio"|"asr_partial"|"phase_change"|"interview_end"|"error", payload, seq}`；断线重连以 `seq` 续传。

### 6.5 非功能需求

- **性能**：落地页 LCP < 2s；面试官首 token < 2s；ASR 转写延迟 < 800ms；报告生成 < 60s（过场页展示进度）。
- **安全与隐私**：简历/音频私有存储（签名 URL）；密码 bcrypt；接口限流；LLM 输入防注入（简历内容作为数据而非指令处理）；提供「删除我的全部数据」入口。
- **可靠性**：面试会话状态持久化（Redis + DB），前端断线重连恢复；批改任务队列化（失败重试）。
- **可观测**：会话级 trace（每场面试的 LLM 调用链）、Token 成本统计。
